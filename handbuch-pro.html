SafeShare Pro Handbuch

Pro ist nicht „ein weiterer Cleaner“. Pro ist ein Policy-Werkzeug: Du definierst Regeln, siehst Änderungen, und schützt funktionale/affiliate Parameter per Whitelist.

Zur Pro App Freischaltung Hilfe
Leitprinzip: begründet entfernen, begründet behalten, nachvollziehbar dokumentieren.

Quickstart (2 Minuten)

  1. Öffne pro-app.html
  2. Füge 1–5 Links ein (jeweils neue Zeile)
  3. Wähle ein Preset (konservativ → aggressiv)
  4. Setze Whitelist-Keys, die bleiben müssen (z. B. ref, tag)
  5. Bereinigen → prüfe im Audit „Entfernt/Behalten“ → kopieren
Profi-Regel: Entfernen ohne Audit ist blind. Pro ist dafür da, dass du beim Publizieren Verantwortung übernehmen kannst.

Pro-Modell: 3 Ebenen

1) Entfernen (Remove)

Typische Tracking-Keys raus: utm_*, gclid, fbclid, msclkid …

2) Behalten (Keep)

Funktionale Keys bleiben: id, q, page, lang …

3) Whitelist (Allowlist)

Affiliate/Partner bewusst behalten: ref, tag, affid …

Audit

Jeder Link liefert: entfernt / behalten / Warnungen. Das ist der Profi-Unterschied.

Whitelist/Allowlist (Praxis)

Eine Whitelist ist nicht „Tracking erlauben“, sondern: zuordnungsrelevante oder funktionale Keys absichern.

Empfohlene Start-Whitelist (Copy/Paste)

ref tag affid affiliate partner partnerid source lang locale id sku q s search page p sort filter variant
Wenn du Publisher bist: nicht pauschal id löschen. Viele Seiten verwenden id für echte Inhalte.

Audit: Was muss sichtbar sein?

  • Removed: Liste aller entfernten Keys (nicht nur der erste Link)
  • Kept: Keys, die vorhanden waren und bewusst bleiben
  • Whitelist-Treffer: Keys, die wegen Allowlist nicht entfernt wurden
  • Warnings: z. B. „unklar: könnte funktional sein“
Für Profis ist Audit wichtiger als „wie viele Keys entfernt“. Audit ist Haftung/Qualität/Verlässlichkeit.

Presets/Policies (professionell, aber übersichtlich)

Du brauchst nicht 20 Modi. Du brauchst 3 klare Policies:

Konservativ

Entfernt nur eindeutig Tracking (utm_*, gclid/fbclid/msclkid). Minimales Risiko.

Standard

Entfernt zusätzlich typische Social/Email-Ballast-Keys (wenn ihr das so definiert). Audit zeigt alles.

Aggressiv

Maximale Kürzung. Nur nutzen, wenn Whitelist sauber gepflegt ist.

Publisher/Team Policy

Ein Preset als „Hausstandard“. Das ist der Pro-Use-Case.

Wenn dir der Preset-Bereich zu unübersichtlich ist: reduziere auf genau diese 3 + 1 Team-Policy. Keine Dropdown-Hölle.

Praxisfälle (kurz)

  • Affiliate: tag/ref muss bleiben → Whitelist.
  • Artikel-Links: utm_* raus, sonst bleibt alles → konservativ.
  • Shop-Links: variant/sku kann nötig sein → audit checken.
  • Plattform-IDs: gclid/fbclid raus → Standard.

Troubleshooting

  • „Neue Version sehe ich nicht“ → Status → Update aktivieren / Cache leeren
  • „Link kaputt nach Bereinigung“ → Preset konservativer + funktionale Keys in Allowlist
  • „Nur erster Link bereinigt“ → Pro App muss Multi-Line korrekt iterieren (Fix im Cleaner-Script)
Wenn du willst, liefere ich dir als nächsten Schritt den konkreten Fix, damit die Pro App alle Links in der Liste verarbeitet (nicht nur den ersten).
